## 1. プロジェクト概要

### 1.1 プロジェクト名
北海道 道の駅コレクション（仮）

### 1.2 目的
北海道の道の駅（約130箇所）を地図で探し、訪問記録をスタンプラリー形式で残せるWebアプリ。ドライブ旅行者が道の駅を効率的に巡り、達成感を得られる体験を提供する。

### 1.3 対象ユーザー
- **未ログインユーザー**: 道の駅の閲覧・検索のみ
- **ログインユーザー**: 訪問記録、お気に入り登録、Tips投稿（Phase 3〜）が可能
- **管理者**: 道の駅データの管理（CRUD）

### 1.4 権限マトリクス

| ユーザー種別 | 閲覧（地図・一覧・詳細） | 訪問記録・お気に入り | Tips投稿（Phase 3〜） | 管理（道の駅データCRUD） |
|-------------|------------------------|---------------------|----------------------|----------------------|
| 未ログインユーザー | ○ | × | × | × |
| ログインユーザー | ○ | ○ | ○ | × |
| 管理者 | ○ | ○ | ○ | ○ |

### 1.5 スコープ

**Phase 1（MVP）:**
- 地図表示・検索（Leaflet + OpenStreetMap）
- 道の駅一覧・詳細ページ
- 訪問記録（スタンプラリー）+ メモ + GPS連携
- ユーザー認証（Google OAuth + Email）
- マイページ（達成率・訪問履歴）
- お気に入り / 巡りたいリスト
- エリア制覇バッジ・マイルストーン
- 道の駅データのローカルキャッシュ（基本的なオフライン対応）

**Phase 2（エンゲージメント強化）:**
- 写真記録
- ルート計画
- 情報修正リクエスト（クラウドソーシング的データ鮮度維持）

**Phase 3（拡張機能）:**
- 道の駅Tips（スタンプラリー特化の構造化レビュー）
- ランキング・統計
- SNS連携

**対応しない範囲:**
- 北海道以外の道の駅（将来的に全国展開の可能性あり）
- ネイティブアプリ（PWA対応は将来検討）

### 2.2 機能詳細

#### 機能1: 地図表示・検索
- **概要**: 北海道の道の駅を地図上にピンで表示し、検索・フィルタリングが可能
- **ユーザーストーリー**: ドライバーとして、近くの道の駅を地図で素早く見つけたい。なぜなら効率的にドライブルートを組みたいから。
- **受け入れ条件**:
  - [ ] Leaflet + OpenStreetMap で北海道の地図を表示
  - [ ] 約130箇所の道の駅をピンで表示
  - [ ] ピンタップで概要カード（名前・住所・写真）を表示
  - [ ] 現在地取得ボタンで近くの道の駅をハイライト
  - [ ] エリア（道東・道北・道央・道南）でフィルタリング
  - [ ] キーワード検索（道の駅名、住所）
  - [ ] ログインユーザーは訪問済み/未訪問/お気に入りでピンの色・形を区別
- **設計メモ**: 振興局データはDBに保持するが、UIでは「道東・道北・道央・道南」の4エリア＋地図ベースの絞り込みを主にする。振興局での詳細フィルタは展開時のオプション。

#### 機能2: 道の駅一覧・詳細
- **概要**: 道の駅をカード形式で一覧表示し、詳細ページで基本情報を閲覧
- **ユーザーストーリー**: 旅行計画者として、道の駅の営業時間や設備を事前に確認したい。なぜなら立ち寄り先を計画的に選びたいから。
- **受け入れ条件**:
  - [ ] カード形式の一覧表示（写真・名前・エリア・訪問済みバッジ）
  - [ ] 詳細ページに基本情報（住所・営業時間・定休日・電話番号・公式URL）
  - [ ] 詳細ページにアクセス情報、設備情報（トイレ・EV充電・Wi-Fi等）
  - [ ] SSG/ISR で事前生成（SEO対策）

#### 機能3: 訪問記録（スタンプラリー）
- **概要**: 道の駅を訪問した際に記録し、GPS連携の有無でバッジの種類を分けることでスタンプラリーの信頼性と達成感を両立
- **ユーザーストーリー**: ドライブ旅行者として、訪問した道の駅を記録して達成感を得たい。なぜなら全制覇を目指すモチベーションになるから。
- **受け入れ条件**:
  - [ ] 「訪問した！」ボタンで訪問日を記録
  - [ ] 訪問時にメモ（テキスト）を入力可能（任意）
  - [ ] 位置情報の任意送信に対応（Geolocation API）
  - [ ] GPS付き訪問（道の駅から半径1km以内）にはゴールドバッジを表示
  - [ ] 自己申告訪問（GPS未送信/範囲外）にはシルバーバッジを表示
  - [ ] マイページで達成率（◯/130）を表示
  - [ ] 地図上で訪問済み（ゴールド/シルバー）/未訪問をビジュアルで区別
- **設計メモ**: GPS認証は強制しない。位置情報を送信しなくても訪問記録は可能。ゴールド/シルバーの区別でゲーム性と手軽さを両立する。

#### 機能4: ユーザー認証
- **概要**: Supabase Auth を利用した認証
- **受け入れ条件**:
  - [ ] Google OAuth でログイン可能
  - [ ] メール + パスワードでログイン可能
  - [ ] 未ログインでも閲覧・検索は可能
  - [ ] ログインユーザーのプロフィール（表示名・アバター）管理

#### 機能5: マイページ
- **概要**: ログインユーザーの訪問記録と達成状況を表示
- **受け入れ条件**:
  - [ ] 訪問済み道の駅の一覧表示（ゴールド/シルバー区別）
  - [ ] 達成率（◯/130）をプログレスバーで表示
  - [ ] 訪問履歴をタイムライン/リスト表示（メモ付き）
  - [ ] エリア制覇バッジ・マイルストーンの表示（機能7と連動）
  - [ ] お気に入りリストの表示（機能6と連動）

#### 機能6: お気に入り / 巡りたいリスト（Phase 1）
- **概要**: 行きたい道の駅をリスト化し、ドライブ計画の下準備ができる。Phase 2のルート計画機能の基盤。
- **ユーザーストーリー**: ドライブ計画者として、次に巡りたい道の駅をリストにまとめたい。なぜなら旅行前に行き先を整理したいから。
- **受け入れ条件**:
  - [ ] 道の駅詳細ページ・一覧・地図からお気に入りに追加/解除
  - [ ] マイページにお気に入りリストを表示
  - [ ] 地図上でお気に入りの道の駅を区別表示（ピンの色・アイコン）
  - [ ] お気に入りの並び替え（追加順/エリア順）

#### 機能7: ゲーミフィケーション（Phase 1）
- **概要**: 130箇所の長い道のりでモチベーションを維持するための段階的な達成感の仕組み
- **ユーザーストーリー**: スタンプラリー参加者として、小さな達成を積み重ねて全制覇を目指したい。なぜなら130箇所は遠い目標なので途中のご褒美が欲しいから。
- **受け入れ条件**:
  - [ ] エリア制覇バッジ: 道東/道北/道央/道南の全道の駅を訪問するとバッジを獲得
  - [ ] マイルストーンバッジ: 10, 30, 50, 100, 130箇所達成時にバッジを獲得
  - [ ] 訪問ペースの可視化: 「今月◯箇所訪問」の表示
  - [ ] マイページにバッジ一覧を表示（獲得済み/未獲得）

#### 機能8: オフラインデータキャッシュ（Phase 1）
- **概要**: 北海道の電波不安定エリアでも道の駅情報を閲覧できるよう、基本データをローカルにキャッシュ
- **ユーザーストーリー**: ドライブ中のユーザーとして、山間部でも道の駅の情報を確認したい。なぜなら北海道は電波が届かない場所が多いから。
- **受け入れ条件**:
  - [ ] 道の駅一覧データ（名前・住所・座標・営業時間・設備）をローカルにキャッシュ
  - [ ] Service Worker でオフライン時にもキャッシュ済みデータを閲覧可能
  - [ ] オンライン復帰時にデータを自動更新
  - [ ] オフライン時の書き込み（訪問記録等）は不可。オンライン必須のUI表示
- **設計メモ**: 完全なPWAは不要。next-pwa 等で Service Worker を導入し、APIレスポンスのキャッシュを優先。オフライン対応は閲覧のみに絞り、書き込みの同期処理（コンフリクト処理・データ整合性）は将来検討。地図タイルのオフラインキャッシュは Phase 3 以降の検討事項。

#### 機能9: 写真記録（Phase 2）
- **概要**: 訪問時に写真を記録し、旅の思い出をビジュアルで振り返れる
- **受け入れ条件**:
  - [ ] 訪問記録に写真（最大3枚）を添付可能
  - [ ] 画像は Supabase Storage に保存
  - [ ] 自分の訪問履歴をタイムライン形式で表示（写真付き）
  - [ ] 画像のリサイズ・最適化を行う（クライアント側で圧縮後アップロード）

#### 機能10: ルート計画（Phase 2）
- **概要**: 複数の道の駅を選択してドライブルートを計画できる。お気に入りリスト（機能6）との連携。
- **受け入れ条件**:
  - [ ] お気に入りリストまたは地図上から複数の道の駅を選択
  - [ ] 選択した道の駅を巡るドライブルートを生成
  - [ ] ルートの所要時間・距離を表示
  - [ ] ルートを地図上に描画
  - [ ] ルートの保存・共有

#### 機能11: 情報修正リクエスト（Phase 2）
- **概要**: ユーザーからの「情報が違う」通報を受け付け、クラウドソーシング的にデータの鮮度を維持する
- **ユーザーストーリー**: 訪問者として、営業時間が変わっていたことを報告したい。なぜなら次に来る人が困らないようにしたいから。
- **受け入れ条件**:
  - [ ] 道の駅詳細ページに「情報が違う」通報ボタンを設置
  - [ ] 通報時に修正対象（営業時間/定休日/電話番号/設備/その他）を選択
  - [ ] 正しい情報と補足コメントを入力可能
  - [ ] 管理者は通報一覧を確認し、承認/却下できる

#### 機能12: 道の駅Tips（Phase 3）
- **概要**: Google Mapsレビューとの差別化として、スタンプラリー文脈に特化した構造化情報を収集
- **設計メモ**: 一般的な5段階レビューではなく、スタンプラリー参加者に役立つ構造化データを集める。130箇所×少数ユーザーでもスカスカに見えない設計。
- **受け入れ条件**:
  - [ ] カテゴリ別のTips投稿（名物グルメ/駐車場/スタンプ台の場所/おすすめ撮影スポット/注意事項）
  - [ ] 各Tipsに「参考になった」ボタン
  - [ ] 道の駅詳細ページにTips一覧を表示
  - [ ] 不適切なTipsの通報機能

#### 機能13: ランキング・統計（Phase 3）
- **概要**: ユーザーの訪問実績や道の駅の人気度をランキング・統計で可視化
- **受け入れ条件**:
  - [ ] 訪問数ランキング（ユーザー別）を表示
  - [ ] 人気の道の駅ランキング（訪問数順）を表示
  - [ ] 人気の道の駅ランキング（Tips評価順）を表示
  - [ ] エリア（道東/道北/道央/道南）別の制覇率を表示

#### 機能14: SNS連携（Phase 3）
- **概要**: 訪問記録やスタンプカードをSNSにシェアし、アプリの認知拡大を図る
- **受け入れ条件**:
  - [ ] 訪問記録をSNS（X/Twitter等）にシェア可能
  - [ ] スタンプカード（達成状況）をシェア可能
  - [ ] エリア制覇バッジをシェア可能
  - [ ] OGP画像を自動生成

## 3. 非機能要件

### 3.1 パフォーマンス
- 地図の初期表示: 2秒以内
- 道の駅データは SSG/ISR で事前生成
- 画像は最適化して配信

### 3.2 セキュリティ
- Supabase RLS（Row Level Security）でアクセス制御
- 入力値のバリデーション（Zod）
- 環境変数による機密情報管理
- HTTPS必須

### 3.3 可用性
- Vercel + Supabase のマネージドインフラで可用性を確保
- 未ログインユーザーはサーバー障害時でもSSG/ISRキャッシュから閲覧可能
- Service Worker による基本的なオフラインアクセス（Phase 1）

### 3.4 SEO
- 道の駅詳細ページは SSG で生成し、検索エンジンからの流入を狙う
- 適切なメタデータ・OGP設定

### 3.5 アクセシビリティ
- キーボード操作対応
- 適切なalt属性
- セマンティックHTML

### 3.6 レスポンシブ対応
- PC・スマホ両対応（レスポンシブデザイン）
- モバイルファーストで設計（ドライブ中のスマホ利用を想定）
- 地図操作はタッチ・マウス両対応

## 7. 制約・前提条件

### 7.1 ビジネス制約
- 対象エリアは北海道のみ（約130箇所）
- 道の駅データは初期シード投入（国交省オープンデータ等を活用）
- 無料枠で運用可能な構成（Supabase Free + Vercel Hobby）

### 7.2 技術的制約
- 地図はOpenStreetMap（Google Maps APIは使用しない）
- Supabase の無料枠の制限（DB 500MB、Storage 1GB、月間5万MAU）
- 画像アップロードは Phase 2 以降
- GPS精度は端末依存（位置情報の半径判定は1kmを基準）

## 11. 今後の検討事項

- **道の駅データの取得方法**: 国交省オープンデータ or スクレイピング or 手動登録
- **GPS判定半径の調整**: 1kmが適切か、ユーザーテストで検証
- **画像のリサイズ・最適化**: クライアント側圧縮（Phase 2で対応予定）
- **地図タイルのオフラインキャッシュ**: Phase 3以降で完全オフライン地図を検討
- **PWA対応の深化**: インストールプロンプト、プッシュ通知
- **全国展開時のデータ構造拡張性**: area_group の都道府県版への拡張

## 12. フェーズ移行の判断基準

### Phase 1 → Phase 2
- MVP公開後、アクティブユーザー数が一定数に達した
- ユーザーから写真投稿・ルート計画の要望が多い
- MVPの主要バグが解消されている

### Phase 2 → Phase 3
- Phase 2機能の利用率が安定している
- Tips投稿に十分なユーザー基盤がある
- ランキング・統計が意味を持つデータ量に達している
