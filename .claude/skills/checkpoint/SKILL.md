# チェックポイント保存

作業の進捗や学んだことを `notes/` ディレクトリに保存し、プロジェクトの「記憶」として蓄積します。

## 実行手順

### Step 1: 保存タイプの選択

ユーザーに確認：

「チェックポイントを保存します。種類を選んでください：
1. **進捗** - 作業の進捗状況（`notes/progress/`）
2. **学習** - 学んだこと・ハマりポイント（`notes/learnings/`）
3. **決定** - 技術的な意思決定（`notes/decisions/`）」

---

### Step 2A: 進捗の保存

ファイル名: `notes/progress/YYYY-MM-DD-[タスク名].md`

```markdown
# 進捗: [タスク名]

**日付**: YYYY-MM-DD
**ブランチ**: `feature/xxx`
**ステータス**: 進行中 / 完了 / ブロック中

## 概要
[何をしているか]

## 完了した作業
- [x] [完了項目1]
- [x] [完了項目2]

## 残りの作業
- [ ] [未完了項目1]
- [ ] [未完了項目2]

## 変更ファイル
- `path/to/file1` - [変更内容]
- `path/to/file2` - [変更内容]

## 次のステップ
1. [次にやること]

## メモ
[気づいたこと、注意点など]
```

---

### Step 2B: 学習の保存

ファイル名: `notes/learnings/YYYY-MM-DD-[トピック].md`

```markdown
# 学習: [トピック]

**日付**: YYYY-MM-DD
**カテゴリ**: バグ / パフォーマンス / 設計 / ツール / その他

## 問題・状況
[何が起きたか、何を調べたか]

## 原因・発見
[なぜそうなったか、何がわかったか]

## 解決策・対処法
[どう解決したか]

## 今後の注意点
- [同じ問題を避けるために]
- [CLAUDE.mdに追加すべきルールがあれば]

## 参考リンク
- [関連URL]
```

---

### Step 2C: 決定の保存

ファイル名: `notes/decisions/YYYY-MM-DD-[決定内容].md`

```markdown
# 決定: [決定内容]

**日付**: YYYY-MM-DD
**ステータス**: 提案 / 承認 / 却下 / 保留

## コンテキスト
[なぜこの決定が必要になったか]

## 検討した選択肢

### 選択肢A: [名前]
- メリット: [利点]
- デメリット: [欠点]

### 選択肢B: [名前]
- メリット: [利点]
- デメリット: [欠点]

## 決定
[選択肢X] を採用する。

## 理由
[なぜその選択をしたか]

## 影響
- [この決定による影響]

## 関連
- [関連するファイル、決定、Issue等]
```

---

### Step 3: 保存の実行

1. `notes/` ディレクトリがなければ作成
2. 適切なサブディレクトリに保存
3. 保存完了を報告

```
「チェックポイントを保存しました。
ファイル: notes/progress/2024-01-15-user-auth.md

このファイルはgitにコミットすることで、チームで共有できます。」
```

---

## ディレクトリ構造

```
notes/
├── progress/          # 作業進捗
│   ├── 2024-01-15-user-auth.md
│   └── 2024-01-16-api-refactor.md
├── learnings/         # 学んだこと
│   ├── 2024-01-15-n-plus-one.md
│   └── 2024-01-16-cache-invalidation.md
└── decisions/         # 技術決定
    ├── 2024-01-10-auth-method.md
    └── 2024-01-12-database-choice.md
```

---

## 使用例

```
/checkpoint
```

## 活用のコツ

- **PR作成時に実行** - 変更の背景を記録
- **問題解決後に実行** - 学びを蓄積
- **設計判断時に実行** - 決定理由を残す
- **定期的に振り返り** - 蓄積された知見を活用
