# セキュリティルール（共通）

このルールは全てのプロジェクトで適用されるセキュリティ基準です。

---

## OWASP Top 10 対策

### 1. インジェクション対策
- SQLはプレースホルダ/プリペアドステートメントを使用
- OSコマンド実行時は引数をエスケープ
- LDAPやXPathも同様に対策

### 2. 認証の不備対策
- パスワードは強力なハッシュアルゴリズムで保存（bcrypt, Argon2）
- セッションIDは認証後に再生成
- ブルートフォース対策（レート制限、アカウントロック）

### 3. 機密データの露出対策
- 通信はHTTPS必須
- 機密情報はログに出力しない
- 不要なデータは保持しない

### 4. XXE対策
- XML外部エンティティの処理を無効化

### 5. アクセス制御の不備対策
- 全てのAPIエンドポイントで認可チェック
- 直接オブジェクト参照の防止（IDOR対策）

### 6. セキュリティ設定のミス対策
- デフォルト認証情報は変更
- 不要なサービス・ポートは無効化
- エラーメッセージに内部情報を含めない

### 7. XSS対策
- 出力は必ずエスケープ
- Content-Security-Policyヘッダーの設定
- HttpOnly, Secure属性のCookie設定

### 8. 安全でないデシリアライゼーション対策
- 信頼できないデータのデシリアライズは避ける

### 9. 既知の脆弱性対策
- 依存パッケージは定期的に更新
- セキュリティアドバイザリを監視

### 10. 不十分なロギング・監視対策
- 認証イベントはログに記録
- 不審なアクティビティを検知する仕組み

## コードレビュー時のセキュリティチェックリスト

```markdown
## セキュリティチェック

### 入力検証
- [ ] ユーザー入力のバリデーション実装
- [ ] ファイルアップロードの検証（拡張子、MIMEタイプ、サイズ）

### 認証・認可
- [ ] 認証が必要なエンドポイントは保護
- [ ] 適切な権限チェック

### データ保護
- [ ] 機密情報のハードコーディングなし
- [ ] ログに機密情報を出力していない

### 出力エンコーディング
- [ ] HTML出力のエスケープ
- [ ] JSONレスポンスの適切なContent-Type
```
