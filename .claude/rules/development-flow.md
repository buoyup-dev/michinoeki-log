# 開発フロールール

全プロジェクト共通の開発フローと規約です。

---

## 開発フロー

### 1. 要件確認フェーズ
- ユーザーの要求を明確化
- 既存コードベースの構造を把握
- 影響範囲を特定

### 2. 設計フェーズ
- 実装方針を検討
- 必要に応じてPlan Modeで計画を作成
- ユーザーに確認を取る

### 3. 実装フェーズ
- 小さな単位で実装・テスト
- 既存のコード規約に従う
- セキュリティを意識

### 4. レビュー・テストフェーズ
- テストを実行して全てパスすることを確認
- リンターでコード品質をチェック
- 必要に応じてcode-reviewerサブエージェントを活用

### 5. コミット・PR作成フェーズ
- 意味のある単位でコミット
- PRは目的と変更内容を明記
- **PRのマージ先は必ず `develop` ブランチを指定する**
  - `main` への直接マージは禁止
  - `gh pr create` 実行時は必ず `--base develop` を付ける
  - PR作成前に「マージ先は develop でよいですか？」とユーザーに確認する

## ブランチ戦略

| プレフィックス | 用途 | 例 |
|--------------|------|-----|
| `feature/` | 新機能開発 | `feature/user-authentication` |
| `fix/` | バグ修正 | `fix/login-validation` |
| `refactor/` | リファクタリング | `refactor/extract-service` |
| `docs/` | ドキュメント | `docs/api-readme` |
| `release/` | リリース準備 | `release/1.2.0` |

## コミットメッセージ規約

```
<emoji> <type>: <subject>

<body>（任意）
```

### Type一覧

| 絵文字 | Type | 用途 |
|-------|------|------|
| ✨ | `feat` | 新機能 |
| 🐛 | `fix` | バグ修正 |
| ♻️ | `refactor` | リファクタリング（機能変更なし） |
| ✅ | `test` | テスト追加・修正 |
| 📝 | `docs` | ドキュメント |
| 🔧 | `chore` | ビルド・設定変更 |
| 🎨 | `style` | コード整形（機能変更なし） |
| ⚡ | `perf` | パフォーマンス改善 |

### 例
```
✨ feat: ユーザー認証機能を追加

- JWT トークンによる認証を実装
- ログイン/ログアウトAPIを追加
```

```
🐛 fix: ログイン時のバリデーションエラーを修正
```

## コードレビュー観点

### 可読性
- 関数名・変数名は意図が明確か
- 過度に長い関数はないか（目安: 50行以内）
- ネストが深すぎないか（目安: 3段階以内）

### 保守性
- 重複コードはないか
- 適切に責務が分離されているか
- マジックナンバーは定数化されているか

### パフォーマンス
- N+1問題はないか
- 不要なループ処理はないか
- 適切なインデックスが設定されているか

## CLAUDE.md 自己改善ルール

Claudeが同じミスを繰り返さないよう、ルールを継続的に改善する。

### ミス発見時のフロー

1. **問題を特定** - 何が間違っていたか明確にする
2. **根本原因を分析** - なぜそのミスが起きたか
3. **CLAUDE.mdを更新** - 再発防止のルールを追記

```
「このミスを二度としないように、CLAUDE.mdを更新して」
```

### 更新の指針

- **具体的に書く** - 曖昧なルールは効果がない
- **例を添える** - Good/Badの例があると効果的
- **既存ルールと整合** - 矛盾するルールを追加しない

### 例

```markdown
## よくあるミス（プロジェクト固有）

### 1. ○○の処理で△△を忘れる
❌ Bad: [問題のあるコード例]
✅ Good: [正しいコード例]
```

## 進捗・学習の記録

### notesディレクトリの活用

プロジェクトルートに `notes/` ディレクトリを作成し、以下を記録：

```
notes/
├── decisions/       # 技術的な意思決定
├── learnings/       # 学んだこと・ハマりポイント
└── progress/        # 作業進捗のチェックポイント
```

### 記録するタイミング

- **PR作成時** - 変更の背景・意図を記録
- **問題解決時** - 原因と解決策を記録
- **設計判断時** - 選択肢と決定理由を記録

これにより、後から参照可能な「プロジェクトの記憶」が蓄積される。
