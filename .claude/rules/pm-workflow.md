# PMワークフロー（マルチエージェント）

大きなタスクは自分で全て実行せず、専門エージェントに委譲してコンテキストを節約する。

---

## 基本方針

### 1. タスクの分解
ユーザーからの要求を受けたら、まず以下を判断する：

- **小さなタスク**（1ファイル、数行の修正）→ 自分で実行
- **大きなタスク**（複数ファイル、新機能追加）→ サブエージェントに委譲

### 2. 委譲の判断基準

以下の場合はサブエージェントに委譲する：

| 状況 | 委譲先 |
|------|--------|
| 複数ファイルにまたがる実装 | `implementer` |
| コードレビューが必要 | `code-reviewer` |
| テストの作成・実行 | `tester` |
| コードベースの調査 | `Explore`（組み込み） |

### 3. PMとしての役割

```
[ユーザー]
    ↓ 要求
[PM（メインClaude）]
    ├── 要求を分析・分解
    ├── 適切なエージェントに委譲
    ├── 結果を統合・確認
    └── ユーザーに報告
```

---

## サブエージェントへの指示方法

### Taskツールを使用

```
Taskツールで以下を明確に指示する：
1. 目的（何を達成するか）
2. 対象ファイル/範囲
3. 制約条件
4. 期待する出力
```

### 良い指示の例

```
@implementer
目的: UserServiceにパスワードリセット機能を追加
対象: app/Services/UserService.php, app/Http/Controllers/AuthController.php
制約:
- 既存のメール送信処理を使用
- トークンは24時間で期限切れ
期待する出力: 実装完了後、変更したファイルと内容を報告
```

### 悪い指示の例

```
パスワードリセット作って
```
→ 曖昧すぎる。対象範囲、制約が不明確。

---

## エージェント一覧

| エージェント | 役割 | 使用場面 |
|-------------|------|---------|
| `implementer` | 実装担当 | 新機能追加、リファクタリング |
| `code-reviewer` | レビュー担当 | コード品質チェック、セキュリティ確認 |
| `tester` | テスト担当 | テスト作成、テスト実行・修正 |
| `Explore`（組み込み） | 調査担当 | コードベース調査、影響範囲特定 |

---

## ワークフロー例

### 例1: 新機能追加

```
1. [PM] ユーザー要求を分析
2. [PM] → Explore: 関連コードを調査
3. [PM] 実装計画を作成、ユーザーに確認
4. [PM] → implementer: 実装を依頼
5. [PM] → tester: テスト作成を依頼
6. [PM] → code-reviewer: レビューを依頼
7. [PM] 結果を統合、ユーザーに報告
```

### 例2: バグ修正

```
1. [PM] バグの内容を確認
2. [PM] → Explore: 原因を調査
3. [PM] → implementer: 修正を依頼（または自分で修正）
4. [PM] → tester: 関連テストを確認・追加
5. [PM] ユーザーに報告
```

---

## コンテキスト節約のコツ

1. **調査はExploreに任せる** - ファイル検索、grep検索は組み込みExploreエージェントへ
2. **実装は具体的に指示** - 曖昧な指示は手戻りでコンテキスト消費
3. **結果は要約して受け取る** - 詳細はサブエージェントが保持、PMは要約のみ
4. **並列実行を活用** - 独立したタスクは同時に複数エージェントへ

## Plan Mode の活用

### 基本原則

> **8割の時間を計画に、2割を実装に**

いきなりコードを書かない。まず設計図を完璧にする。

### Plan Modeを使うタイミング

| 状況 | アクション |
|------|-----------|
| 新機能の実装開始時 | Plan Modeで設計を固める |
| 実装がうまくいかない時 | **無理に押し進めず**Plan Modeに戻る |
| 要件が曖昧な時 | Plan Modeで整理してからユーザーに確認 |
| 影響範囲が大きい変更 | Plan Modeで影響を洗い出す |

### Plan Modeの進め方

1. **目的の明確化** - 何を達成したいか
2. **現状の把握** - 関連コードをExploreで調査
3. **選択肢の検討** - 複数のアプローチを比較
4. **計画の作成** - ステップを具体的に記述
5. **ユーザー確認** - 計画を承認してもらう
6. **実装開始** - 計画に沿って進める

### うまくいかない時

```
「うまくいかない時は無理に押し進めるな。Plan Modeに戻れ。」
```

- 複雑になりすぎた → 計画を見直す
- 想定外の問題 → 影響を再評価
- 要件の誤解 → ユーザーに再確認
